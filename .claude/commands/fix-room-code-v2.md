# 指令：彻底修复“连接码无效或已过期”（v2）

## 目标
修正信令服务器中房间创建/加入的逻辑，确保共享方生成的 6 位短码能被对方成功加入，不再返回“连接码无效或已过期”。

## 背景
当前现象：  
- 共享方启动后生成短码 `B3B95R`  
- 加入方发送 `{"type":"join", "roomId":"B3B95R"}`  
- 信令服务器回复 `{"type":"error","message":"连接码无效或已过期"}`  

## 需要检查并修复的点

1. **字段名统一**  
   - 搜索所有处理消息的代码（如 `signaling-server.js`、`server.js`），找出创建房间和加入房间时使用的字段。  
   - 常见错误：创建时用了 `code`，加入时用了 `roomId`；或者使用了 `msg.room` / `msg.id` 等不一致的名称。  
   - **必须统一为 `roomId`**，且字符串严格匹配。

2. **房间存储逻辑**  
   - 检查创建房间时是否正确执行了 `rooms.set(roomId, ...)`。  
   - 确保 `roomId` 值取自消息体中的正确字段，并进行 `.trim()` 去除首尾空格。  
   - 加入房间时，使用 `rooms.has(roomId)` 检查，**切勿**使用额外的过期判断或自动清理机制（除非房间内所有成员断开）。

3. **移除任何隐式过期检查**  
   - 搜索代码中是否存在 `setTimeout`、`Date.now()`、`timestamp` 等用于判断房间是否过期的逻辑，如有则**全部移除**或注释掉。房间仅当房主关闭共享时手动销毁。

4. **增强调试输出**  
   - 在创建房间后添加：`console.log('[信令] 房间已创建:', roomId, '当前房间列表:', Array.from(rooms.keys()));`  
   - 在收到加入请求后添加：`console.log('[信令] 收到加入请求:', msg.roomId, '房间存在:', rooms.has(msg.roomId));`  
   - 这样可以从控制台直接看出问题所在。

5. **前端一致性检查**  
   - 确认前端生成短码后，发送的创建消息格式为 `{ type: 'create', roomId: 'xxx' }`。  
   - 确认加入时消息格式为 `{ type: 'join', roomId: 'xxx' }`，且 `roomId` 取自输入框（已 `trim()`）。  
   - 复制短码功能应直接复制原始字符串，不添加任何其他文本。

## 验收标准
- 启动双实例（或用 `test-two.bat`），主实例生成短码。  
- 在测试实例输入该短码，控制台输出“房间存在: true”，并成功建立 P2P 连接。  
- 不再出现“连接码无效或已过期”错误提示。  
- 如果测试通过，再与朋友实际测试（朋友需连接到你的局域网 IP 或公网信令地址）。
