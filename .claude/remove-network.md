# 指令：移除网络配置弹窗 + 纯短码一键连接

## 目标
彻底移除点击“开始共享”后出现的局域网共享链接弹窗，实现纯短码连接：主持方生成 6 位短码，加入方输入短码即可连接，**不显示任何 IP 地址、不要求对方修改设置**。

## 执行步骤

### 1. 定位并删除网络配置弹窗组件
- 搜索整个 `src/` 目录，找到包含以下关键字的代码：
  - `pc://`、`局域网共享链接`、`同WiFi`、`需在设置中将信令服务器改为`
- 确定这些内容所在的 UI 组件（可能是 `Host.svelte`、`StartScreenDialog.svelte` 或其他模态框）。
- **完全删除**该弹窗的 HTML 结构和触发逻辑（删除对应的 `<div>`、`{#if}` 块等）。
- 如果弹窗中包含“复制链接”等功能，可以直接移除，因为短码系统已经可以唯一标识房间。

### 2. 修改“开始共享”按钮行为
- 按钮点击事件不再触发弹窗，而是**直接调用屏幕选择 + 信令房间创建**。
- 流程简化为：
点击“开始共享” → 弹出系统屏幕选择窗口（Electron desktopCapturer）→
用户选择共享源 → 自动生成 6 位短码 → 短码显示在主持界面（超大居中，可一键复制）

text
- 如果原先的弹窗是在屏幕选择之前出现的，则移除后流程更顺畅。

### 3. 修复信令地址逻辑，彻底杜绝 IP 泄露
- 确保**信令地址永远不需要用户手动输入**。
- 加入方使用固定的信令地址 `ws://localhost:3456`（本机直接连接）或通过信令服务器转发的机制，不需要知道主持方的 IP。
- 如果是局域网场景，信令服务器监听 `0.0.0.0:3456`，加入方只需知道信令服务器的 IP（之前已实现自动获取），但这个 IP 应该通过底层自动传递，**绝不在 UI 上显示**。
- 检查 `src/lib/webrtc.js` 或 `src/renderer/src/WebRTC.svelte`，确保 WebSocket 连接地址不再来自 UI 输入，而是直接使用 `ws://${自动获取的主持方IP}:3456`（由主进程 IPC 提供），对于加入方，这个 IP 应该由短码系统内部解析或由信令服务器中转，而不是让用户看到。
- 理想方案：加入方输入短码后，信令服务器根据短码找到主持方，并返回主持方的信令地址（或直接由信令服务器担任中间人），不需要用户知道主持方 IP。检查现有代码是否已支持，若不支持，则修改信令逻辑：当加入方发送 `join` 消息时，信令服务器将主持方的信令地址包含在回复中，或者直接由信令服务器转发所有后续 WebRTC 信令，不依赖客户端直接连接主持方信令服务器。

### 4. 清理设置界面中的网络配置残留
- 打开设置面板组件，移除所有“信令服务器地址”、“端口”等输入框。
- 确保设置面板仅包含皮肤选择、摄像头/麦克风设备选择等非网络项。

### 5. 验证
- 启动双实例测试 (`test-two.bat`)。
- 主持方点击“开始共享”，应直接弹出屏幕选择窗口，不再有任何 IP 弹窗。
- 选择窗口后，主持界面显示短码（如 `ZNDRAN`），并有一键复制按钮。
- 加入方输入短码，点击加入，直接开始连接，无需填写任何信令地址。
- 连接成功后，双方能看到共享画面，功能正常。
- 检查设置面板，确认无网络配置项。

## 重要约束
- **仅修改脚本逻辑和 HTML 结构，不得修改任何 CSS 样式或布局。**
- 保留所有皮肤切换功能，新增元素需自动适配皮肤变量。
